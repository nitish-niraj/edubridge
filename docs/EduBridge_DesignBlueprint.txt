
EduBridge
Design & Feature Blueprint
Three-Portal Design System  Ę  Student  Ę  Teacher  Ę  Admin

??  Student Portal Ś Bloom Theme??  Teacher Portal Ś Calm & Simple??  Admin Portal Ś Clean & Functional
For use by AI coding agents Ś includes colour tokens, typography, component rules & full feature specs per portal


Design Philosophy Overview
EduBridge has three completely distinct user groups Ś each with radically different needs, ages, and comfort levels with technology. The design system is split into three separate visual identities that share zero visual overlap, so each user feels the platform was built specifically for them.

?? STUDENT PORTAL
Ages 10¢25Bold, playful, Bloom-inspired. Warm corals, sunshine yellow, fresh mint. Bubbly rounded cards. Illustrated micro-details. Energetic and joyful.?? TEACHER PORTAL
Ages 55¢75Extremely simple. Generous whitespace. Sage green + white. Very large text (18px base). No gradients. No animations. Every element labeled clearly.?? ADMIN PORTAL
Platform teamFunctional & efficient. Slate/blue-gray. Data-dense tables. Standard dashboard conventions. Dark sidebar, white content area.

SECTION 2 Ś TEACHER PORTAL  ??  Simple & Calm Design System
Teacher Portal Ś Design Principles
Teachers are 55¢75 years old. Many are using smartphones for the first time. The design must be EXTREMELY simple Ś like a newspaper or a bank passbook. No gradients. No animations. No tiny text. No icon-only buttons. Every element must be labeled. When in doubt, make it bigger and simpler.

??  CRITICAL RULE: No feature on the teacher portal requires more than 2 taps/clicks to complete. If it takes 3 steps, redesign it. Every page has a single clear purpose and a single prominent action button.

4.1  CSS Architecture
Use CSS Custom Properties (variables) scoped per portal. Apply via a class on the root <html> or <body> element.

Feature / ScreenDesign + Build Instructions for AI AgentCSS Scoping StrategyIn app.css: define three root scopes. [data-portal="student"] { --primary: #E8553E; --bg: #FFF8F0; --font: "Nunito", sans-serif; --radius-card: 20px; } [data-portal="teacher"] { --primary: #3D6B4F; --bg: #FAFAF8; --font: "Arial", sans-serif; --radius-card: 10px; } [data-portal="admin"] { --primary: #2563EB; --bg: #F8FAFC; --font: "Inter", sans-serif; --radius-card: 8px; }. In AppLayout.vue: set :data-portal="$page.props.auth.user.role" on the <body> tag.Google Fonts LoadingIn resources/js/app.js or resources/views/layouts/app.blade.php: load fonts conditionally. If student portal: load Fredoka One + Nunito. If teacher portal: load Georgia (system) + Arial (system). If admin portal: load Inter. This prevents loading all 3 font stacks on every page load Ś keeps performance fast.Tailwind ConfigurationExtend tailwind.config.js with custom colours: student: { coral: "#E8553E", yellow: "#F5C518", mint: "#4CB87E", peach: "#FFAB76", bg: "#FFF8F0" }, teacher: { sage: "#3D6B4F", light: "#EDF4F0", bg: "#FAFAF8" }. Create custom utility classes: .card-student (applies student shadow + radius), .card-teacher (applies teacher border + radius), .btn-student-primary, .btn-teacher-primary.Vue Component NamingPrefix all portal-specific components: Student/TeacherCard.vue, Student/BookingModal.vue, Teacher/AvailabilityRow.vue, Admin/UserTable.vue. Shared components (used in multiple portals) live in /components/Shared/: Shared/Avatar.vue, Shared/StarRating.vue, Shared/LoadingSpinner.vue. This makes it immediately clear which portal a component belongs to.Responsive BreakpointsUse Tailwind defaults: sm=640px, md=768px, lg=1024px, xl=1280px. Student portal: mobile-first (most students on phone). Teacher portal: tablet-first (teachers often on iPad or large Android). Admin portal: desktop-first (operations team on laptop/monitor). Write CSS in that priority order for each portal.
